Método Kanban

Kanban es un método de gestión de trabajo que enfatiza la entrega continua sin sobrecargar al equipo de desarrollo. Se originó en Toyota como sistema para controlar la cadena de producción y ha sido ampliamente adoptado en desarrollo de software y otros campos.

Principios Fundamentales

1. Visualizar el trabajo

Hacer visible el trabajo y las reglas para ejecutarlo. Esto ayuda a entender el flujo de trabajo y a identificar cuellos de botella o problemas.

2. Limitar el WIP

Limitar la cantidad de trabajo en progreso (Work In Progress). Esto evita la sobrecarga del equipo y mejora el tiempo de entrega.

3. Gestionar el flujo

Monitorear, analizar y optimizar el flujo de trabajo para hacerlo predecible y para identificar problemas.

4. Hacer políticas explícitas

Definir, publicar y comunicar las reglas y políticas que rigen cómo funciona el proceso.

5. Implementar ciclos de feedback

Establecer mecanismos de retroalimentación para facilitar la colaboración y mejorar el proceso.

6. Mejorar colaborativamente

Trabajar juntos para mejorar continuamente el sistema, usando métodos científicos y modelos compartidos.

Tablero Kanban

El tablero Kanban es la representación visual del flujo de trabajo. Cada columna representa un estado en el proceso, y cada tarjeta representa un elemento de trabajo.

Elementos del tablero Kanban

  • Columnas: Representan estados o etapas del proceso.
  • Tarjetas: Representan elementos de trabajo o tareas.
  • Límites WIP: Número máximo de elementos permitidos en cada columna.
  • Swimlanes: Filas horizontales para categorizar los tipos de trabajo.
  • Políticas: Reglas explícitas sobre cómo se mueve el trabajo entre columnas.

Métricas Kanban

Kanban utiliza diversas métricas para medir el rendimiento del flujo de trabajo y buscar oportunidades de mejora.

Lead Time (Tiempo de Entrega)

Tiempo total desde que se crea hasta que se completa una tarea.

Cómo medirlo:

Fecha de finalización - Fecha de creación

Interpretación:

Un menor lead time indica un proceso más eficiente. La estabilidad del lead time indica un proceso predecible.

Ejemplo: Si una tarea se crea el 1 de enero y se completa el 10 de enero, el lead time es de 9 días.

Cycle Time (Tiempo de Ciclo)

Tiempo de trabajo activo en una tarea, desde que se comienza a trabajar hasta que se completa.

Cómo medirlo:

Fecha de finalización - Fecha de inicio de trabajo

Interpretación:

Mide la eficiencia interna del equipo. Un cycle time menor indica mayor eficiencia en el proceso de trabajo.

Ejemplo: Si se comienza a trabajar en una tarea el 5 de enero y se completa el 10 de enero, el cycle time es de 5 días.

Throughput (Rendimiento)

Cantidad de tareas completadas por unidad de tiempo.

Cómo medirlo:

Número de tarjetas que pasan a "Completado" en un período (día/semana/mes)

Interpretación:

Indica la capacidad del equipo para entregar valor. Un throughput estable facilita la predicción.

Ejemplo: Si un equipo completa 20 tareas en un mes, su throughput es de 20 tareas/mes o aproximadamente 5 tareas/semana.

Cumulative Flow Diagram (CFD)

Visualización de cómo fluyen los elementos a través del sistema a lo largo del tiempo.

Cómo interpretarlo:

  • Bandas estables indican un flujo constante
  • Bandas ensanchadas muestran acumulación (cuellos de botella)
  • La distancia vertical indica el WIP
  • La distancia horizontal aproxima el lead time

Visualización de métricas

Las métricas de Kanban se suelen visualizar en diferentes gráficos:

  • Control Chart: Muestra el cycle time de elementos individuales a lo largo del tiempo, con líneas que marcan la media y desviación estándar.
  • Cumulative Flow Diagram: Muestra la cantidad de elementos en cada estado a lo largo del tiempo, coloreados por estado.
  • Lead Time Distribution: Histograma que muestra la distribución de lead times, útil para analizar la predictibilidad.
  • Throughput Run Chart: Muestra la cantidad de elementos completados por unidad de tiempo.

Prácticas Recomendadas

Políticas Explícitas

Definir y publicar las reglas de operación del sistema Kanban.

  • Definir criterios claros para cada columna
  • Establecer límites WIP y explica por qué
  • Documentar reglas de priorización
  • Definir clases de servicio para diferentes tipos de trabajo
  • Establecer política de "hecho" para cada etapa

Ejemplo:

Política para "En Revisión":

Para mover un elemento a "En Revisión", debe:

  • Tener todos los tests unitarios pasando
  • Tener la documentación actualizada
  • Cumplir con los estándares de código
  • Tener una solicitud de pull/merge creada

Reuniones de Seguimiento

Cadencias de revisión para gestionar el flujo y mejorar el proceso.

  • Daily Standup: Breve reunión diaria para revisar el estado del tablero, identificar bloqueos y planificar el día.
  • Replenishment Meeting: Reunión periódica (semanal/quincenal) para rellenar el backlog con nuevos elementos.
  • Service Delivery Review: Revisión regular del rendimiento del servicio basada en métricas.
  • Operations Review: Revisión de la efectividad de las políticas y procesos, puede ser mensual o trimestral.
  • Risk Review: Revisión de riesgos potenciales y definición de estrategias de mitigación.

Formato para Daily Standup:

Enfoque en el tablero, no en las personas:

  • Empezar desde la derecha (completado) y moverse hacia la izquierda
  • Identificar elementos bloqueados o en riesgo
  • Discutir elementos que no se han movido recientemente
  • Verificar si hay violaciones de los límites WIP

Implementación de Kanban

Pasos para implementar Kanban en un equipo

  1. Visualizar el flujo actual: Comienza mapeando el proceso actual tal como es, no como debería ser.
  2. Identificar los tipos de trabajo: Clasifica el trabajo por tipo, urgencia, o valor para el cliente.
  3. Definir límites WIP: Comienza con un límite conservador y ajusta según sea necesario.
  4. Definir políticas explícitas: Crea acuerdos sobre cómo se gestionará el trabajo.
  5. Implementar reuniones clave: Establece las cadencias de seguimiento y mejora.
  6. Medir y analizar: Comienza a recopilar métricas desde el primer día.
  7. Identificar cuellos de botella: Usa las métricas para encontrar oportunidades de mejora.
  8. Experimentar y evolucionar: Implementa mejoras y evalúa su impacto.

Evolución del sistema Kanban

Un sistema Kanban efectivo evoluciona con el tiempo. Esta evolución puede seguir estas etapas:

Etapa 1: Kanban de Equipo

  • Visualización del flujo de trabajo del equipo
  • Límites WIP básicos
  • Métricas simples
  • Enfoque en la entrega de valor

Etapa 2: Kanban con Servicios

  • Definición de clases de servicio
  • Políticas explícitas detalladas
  • Métricas avanzadas
  • Gestión de flujo activa
  • Optimización continua

Etapa 3: Kanban de Entrega

  • Integración con otros equipos
  • Visualización de dependencias
  • Planificación basada en rendimiento
  • Predicción estadística
  • Optimización del flujo de valor

Etapa 4: Kanban Empresarial

  • Coordinación entre múltiples equipos
  • Gestión de portafolio
  • Modelado de flujo integral
  • Optimización de la cadena de valor
  • Mejora continua a nivel organizacional

Kanban vs Scrum

Kanban y Scrum son dos métodos ágiles populares, pero tienen diferencias significativas en su enfoque y aplicación:

Aspecto Kanban Scrum
Cadencia Flujo continuo Iterations/Sprints fijos
Entrega Continua, según demanda Al final de cada sprint
Roles No prescribe roles específicos Product Owner, Scrum Master, Development Team
Cambios Se pueden hacer en cualquier momento No se recomiendan durante el sprint
Métricas Lead time, cycle time, throughput Velocity, burndown chart
Reuniones Según necesidad, cadencias flexibles Ceremonias predefinidas
WIP Limitado explícitamente Limitado indirectamente por sprint backlog

¿Cuándo usar cada uno?

Escenarios ideales para Kanban

  • Trabajo con prioridades cambiantes
  • Mantenimiento de sistemas
  • Equipos de soporte
  • Entornos de producción continua
  • Proyectos con entregas frecuentes
  • Cuando se busca mejorar un proceso existente

Escenarios ideales para Scrum

  • Desarrollo de productos nuevos
  • Equipos con menos experiencia en ágil
  • Proyectos con alcance definido
  • Cuando se necesita estructura y predictibilidad
  • Equipos que trabajan en múltiples proyectos
  • Cuando se busca cambiar radicalmente la forma de trabajar

Combinando lo mejor de ambos: Scrumban

Scrumban es un enfoque híbrido que combina elementos de Scrum y Kanban:

  • Utiliza la estructura de roles y ceremonias de Scrum
  • Aplica los límites WIP y el flujo continuo de Kanban
  • Puede mantener sprints como cadencias de planificación, no como compromisos de alcance
  • Enfatiza las métricas de flujo de Kanban
  • Facilita la transición de Scrum a un enfoque más fluido
